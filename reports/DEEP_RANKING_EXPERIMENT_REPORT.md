# 固定候选集深度精排实验报告

## 1. 实验目标与协议

本轮完成工程设计中的扩展 3-5：

1. 固定候选集 DeepFM；
2. DIN 序列精排；
3. 完播、强兴趣、短播多任务精排。

所有模型使用完全相同的候选集，避免把召回变化误判为精排收益：

- Validation 训练候选：`R1.0 + R2.4 + R3.6`，每路 Top150；
- Test 评估候选：`R1.0R + R2.4R + R3.6R`，每路 Top150；
- Validation 训练候选最多保留 300 个，Test 候选不按标签采样；
- 训练候选 2,542,650 行，Test 候选 3,515,663 行；
- Test 标签不参与训练、早停或候选筛选；
- 同一 `user-date-video` 的重复观看标签按最大值聚合。

深度模型统一使用 RTX 4060 Ti 8GB、3 epochs、batch size 8192、embedding dim 16。结果文件位于：

- `artifacts/local_8gb_large/candidate_ranking/`
- `artifacts/local_8gb_large/deep_candidate_ranking/`

## 4. 多任务结论

MMoE 同时预测：

- 完播 AUC：`0.810799`；
- 强兴趣 AUC：`0.522291`；
- 短播 AUC：`0.723920`。

多任务共享表示显著优于 DeepFM 和 DIN，但当前强兴趣正样本过稀，加入强兴趣分数会降低主排序指标；短播任务可学习，但直接惩罚短播会提高覆盖率、降低 Recall/NDCG 和 Cold Recall。因此最终采用“多任务训练、完播头排序”作为最佳深度 challenger，不采用业务加权分数替代主排序。

## 5. 失败诊断与工程决策

DeepFM 首跑曾因 PyTorch Embedding 默认初始化尺度过大，导致 FM 二阶项输出饱和，Test AUC 仅 `0.4963`。修复为小尺度正态初始化并清零一阶 embedding 后，AUC 提升至 `0.6586`。首跑结果已作废。

深度模型均使用点式 BCE，而最终目标是用户-天内 Top100 相对排序；这解释了它们即使有较高点式 AUC，仍未超过 LambdaRank。最终部署决策保持：

```text
R1.0R + R2.4R + R3.6R，各 Top150
  -> 合并去重
  -> PR3 LambdaRank
  -> Top100
```

`DR3.mmoe_complete` 保留为最佳深度 challenger 和面试展示模型，不替换 PR3。后续只在新的 Validation 窗口上实验 listwise/pairwise loss 或 OOF 深度分数融合，禁止继续使用当前 Test 选择权重。

## 6. 时间稳定性

所有模型均存在明显逐日退化：

| 模型 | 2020-09-01 Recall@100 | 2020-09-05 Recall@100 |
|---|---:|---:|
| PR3 | 0.304872 | 0.132930 |
| DR1.deepfm | 0.192943 | 0.055492 |
| DR2.din | 0.228555 | 0.064732 |
| DR3.mmoe_complete | 0.288595 | 0.088980 |

因此生产工程必须采用日级或窗口级刷新，并将按天 Recall/NDCG、候选新鲜度分布和冷启动占比作为监控指标。
